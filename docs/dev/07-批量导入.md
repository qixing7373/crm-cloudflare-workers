# 批量导入方案

## 当前架构

批量导入采用“前端解析 + 本地队列 + 后端分片同步”的架构。

1. 前端读取 Excel/CSV。
2. 前端按字段配置做表头匹配、手机号清洗、去重、状态标准化。
3. 前端将清洗后的记录写入浏览器本地 SQLite 队列。
4. 同步引擎每次取一个分片调用 `/api/import/sync`。
5. 后端对该分片分类、写入联系人表、写入导入历史和明细日志。

旧版 `/api/import/upload`、`/api/import/confirm` 已废弃，不再作为当前开发入口。

## 关键限制

| 项 | 当前值 |
| --- | --- |
| 最大文件 | `20MB` |
| 单文件最大清洗记录 | `50000` |
| 前后端同步分片 | `200` |
| 后端写入子批量 | `25` |
| 必须字段 | 手机号 |

`BATCH_SIZE=200` 是当前稳定值。直接升到 `300` 可能触发 Workers/D1 单次请求压力过高，后续要提升速度应优先做并发窗口、重试和进度恢复，而不是单纯增大分片。

## 状态字段

导入行支持可选状态：

| 输入 | 含义 |
| --- | --- |
| 空 | 普通未开发 |
| `undeveloped` | 普通未开发 |
| `developed` | 已开发，入库后归属当前导入用户 |

批量上传已开发数据时，必须在状态列中传 `developed`。未写状态的记录永远按普通未开发处理。

## 后端分类规则

| 场景 | 结果 |
| --- | --- |
| 系统不存在该手机号 | `added` |
| 系统存在未开发，数据有变化 | `updated` |
| 系统存在未开发，数据无变化 | `skipped` |
| 系统存在未开发，导入状态为已开发 | `updated` 并归属当前用户 |
| 系统存在已开发 | `frozen`，不覆盖原归属 |

## 导入历史

导入历史由 `import_log` 和 `contact_log` 组成：

- `/api/import/history`：所有 manager+ 可查看全部导入批次。
- `/api/import/:id`：查看某个批次的新增、更新、重复、冻结明细。
- 超级管理员和子管理员都可以查看所有上传过的新增数据，不再只显示最新几条。

## 前端状态

导入页面应展示：

- 本地解析状态。
- 云端重复 Hash 提醒。
- 当前同步分片进度。
- 新增、更新、跳过、冻结数量。
- 同步失败后的重试入口。
- 导入历史列表和单批明细。

## 性能策略

- 分片大小保持 `200`。
- 写入批量保持 `25`，避免 D1 batch 压力过高。
- 前端本地队列保存进度，页面卡住或刷新后可恢复。
- 后端每个分片只做当前分片内的查询、分类、写入和日志记录。
- 上传卡住优先检查网络错误、Workers 超时、D1 写入错误和本地队列状态。
